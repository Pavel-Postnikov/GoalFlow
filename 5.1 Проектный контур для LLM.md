# Проектный контур для LLM

Как собрать устойчивый контекст, чтобы любой ассистент работал предсказуемо

### ⚠️ Важное уточнение

**Этот урок — ознакомительная информация о концепции проектного контура.**

**Два основных сценария:**

**1. Локальный контекст (работа с кодом):**

- Требуют AI-агентов: **[Claude Code](https://github.com/anthropics/claude-code), [GitHub Copilot](https://github.com/features/copilot), [Cursor](https://cursor.com/), [Cline](https://marketplace.visualstudio.com/items?itemName=saoudrizwan.claude-dev), [GigaCode](https://giga.chat/)**
- Проект = репозиторий с файлами правил (`CLAUDE.md`, `.cursor/rules`)
- Контекст живёт в файлах проекта, ассистент видит код и структуру

**2. Веб-интерфейсы (общая работа с ИИ):**

- Используйте встроенные проекты: **[ChatGPT (проекты)](https://chatgpt.com/), [Claude (проекты)](https://claude.ai/), [Perplexity (пространства)](https://www.perplexity.ai/), [Яндекс GPT (проекты)](https://ya.ru/ai/gpt)**
- Контекст настраивается внутри проекта на сайте через интерфейс
- Не требует установки дополнительных инструментов

*Далее в уроке описывается общая концепция, применимая к обоим сценариям.*

## Зачем нужны проекты

**Проблема обычного промптинга:** каждый раз объясняешь одно и то же. LLM не помнит правила, термины, стиль работы. Копируешь один и тот же контекст в новые чаты.

**Решение:** проект = контейнер контекста, который живёт отдельно от диалога.

**Проект ≠ фича "Projects" в интерфейсе**

Проект — это структура файлов и правил. Она может жить в:

- Папке на компьютере
- Репозитории с кодом (для ИИ-агентов)
- Notion/Obsidian
- Проектах в ChatGPT / Claude / Яндекс GPT (через веб-интерфейс)

Главное — структура. Инструмент вторичен.

## Структура проекта

Проект состоит из 3 частей:

- **1. Правила:** как ассистент должен думать, отвечать, спорить с тобой
- **2. Знания:** документы, термины, примеры, справочники, решения
- **3. Артефакты:** черновики, финальные тексты, чеклисты, лог решений

## Минимальный скелет (папка на компьютере)

Создай папку `project-name/` и 5 файлов:

    project-name/
        ├── 00-brief.md           # цель, аудитория, ограничения, критерии
        ├── 01-rules.md           # как работать, формат, запреты
        ├── 02-knowledge/         # факты, термины, документы
        ├── 03-templates.md       # библиотека промптов (5-15 штук)
        └── 04-decisions-log.md   # что решили и почему

### 00-brief.md

    # Проект: [название]

        ## Цель
        [Зачем делаем]

        ## Аудитория
        [Для кого]

        ## Ограничения
        - Что нельзя делать
        - Что не входит в проект

        ## Критерии качества
        Как понять, что результат готов:
        - [ ] Критерий 1
        - [ ] Критерий 2

### 01-rules.md

    # Правила работы с ассистентом

        ## Формат ответа по умолчанию
        - Структура: тезис → обоснование → пример
        - Длина: не больше 3 абзацев на один пункт

        ## Стиль
        - Без воды, конкретика
        - Термины объясняй в скобках при первом упоминании

        ## Запреты
        - Не выдумывай факты
        - Если данных не хватает — спроси, не додумывай
        - Не давай общие советы, только под наш контекст

        ## Как спорить
        - Если я ошибаюсь — скажи прямо, с примером
        - Предлагай альтернативы, если есть лучший путь

### 02-knowledge/

Папка с документами:

- `glossary.md` — термины проекта
- `examples.md` — 3-5 эталонных примеров
- `references.md` — ссылки на источники
- Любые PDF, txt, md файлы со справками

### 03-templates.md

    # Библиотека промптов проекта

        ## Промпт 1: Проверка черновика
        Задача: найти дыры в тексте
        Input: черновик
        Output: список проблем + предложения

        ## Промпт 2: Генерация примеров
        ...

### 04-decisions-log.md

    # Лог решений

        ## 2024-12-16: Выбрали формат ответов
        **Решение:** структура "тезис → пример → вывод"
        **Почему:** короче читать, легче проверять
        **Проверить:** через неделю посмотреть, удобно ли

## Как переносить в разные LLM

**🌐 Веб-интерфейсы с встроенными проектами**

Эти инструменты позволяют настраивать контекст проекта через веб-интерфейс. Не требуют установки дополнительного ПО — всё настраивается на сайте.

**Контекст загружается через интерфейс: файлы, инструкции, база знаний.**

### ChatGPT (проекты)

#### Что дает

Ассистент видит чаты, файлы и инструкции внутри проекта.

#### Примерная настройка

1.  Создай новый проект в [ChatGPT](https://chatgpt.com/)
2.  Добавь содержимое файла `01-rules.md` в раздел с инструкциями проекта
3.  Загрузи файлы `00-brief.md` и документы из папки `knowledge/`
4.  Добавь 3-5 эталонных примеров
5.  Веди `decisions-log` как отдельный файл

Примечание: Названия разделов и кнопок могут отличаться в зависимости от версии интерфейса.

#### Критично

Контекст живёт только внутри проекта. Чаты из других проектов не видны.

### Claude (проекты)

#### Что дает

Отдельное рабочее пространство со своей историей и базой знаний.

#### Примерная настройка

1.  Создай новый проект в [Claude](https://claude.ai/)
2.  Добавь содержимое файла `01-rules.md` в раздел с инструкциями проекта
3.  Загрузи документы в базу знаний: `00-brief.md`, файлы из папки `knowledge/`, примеры

Примечание: Названия разделов и кнопок могут отличаться в зависимости от версии интерфейса.

#### Критично

**Контекст между чатами не общий автоматически.** Если правило или факт важен, он должен жить в базе знаний, а не только "в одном чате".

### Perplexity (пространства)

#### Что дает

Центр знаний с настраиваемыми инструкциями плюс поиск в вебе и по файлам.

#### Примерная настройка

1.  Создай новое пространство в [Perplexity](https://www.perplexity.ai/)
2.  Загрузи файлы: `00-brief.md` и документы из папки `knowledge/`
3.  Добавь содержимое файла `01-rules.md` в раздел с инструкциями для ИИ

Примечание: Названия разделов и кнопок могут отличаться в зависимости от версии интерфейса.

#### Тактика

Используй режим "спроси у базы" и "дополни из веба" как два разных шага.

### Gemini (гемы)

#### Что дает

Это ближе к "пресету эксперта", чем к папке с файлами: ты сохраняешь детальные инструкции для повторяемых задач.

#### Примерная настройка

1.  Создай новый гем (Gem) в [Google Gemini](https://gemini.google.com/)
2.  Добавь сжатую версию содержимого файла `01-rules.md`
3.  Добавь примеры входов и выходов (2-5)

Примечание: Названия разделов и кнопок могут отличаться в зависимости от версии интерфейса.

#### Знания

Знания чаще подаёшь через промпты или файлы по ситуации (зависит от твоего процесса работы).

### Яндекс GPT (проекты)

#### Что дает

Российская платформа с поддержкой проектов, собственной базой знаний и историей диалогов.

#### Примерная настройка

1.  Создай новый проект в [Яндекс GPT](https://ya.ru/ai/gpt)
2.  Добавь содержимое файла `01-rules.md` в раздел с инструкциями проекта
3.  Загрузи документы: `00-brief.md` и файлы из папки `knowledge/`
4.  Настрой параметры модели под свои задачи

Примечание: Названия разделов и кнопок могут отличаться в зависимости от версии интерфейса.

#### Особенности

Поддерживает работу с документами на русском языке, имеет интеграцию с экосистемой Яндекса. Контекст сохраняется внутри проекта.

## LLM для разработки (код)

**💡 Локальный контекст с AI-агентами**

Для работы с кодом в IDE требуются специальные AI-агенты ([Claude Code](https://github.com/anthropics/claude-code), [Copilot](https://github.com/features/copilot), [Cursor](https://cursor.com/), [Cline](https://marketplace.visualstudio.com/items?itemName=saoudrizwan.claude-dev)). Они работают с локальными файлами и репозиторием, видят структуру проекта и код.

**Контекст настраивается через файлы в репозитории, а не через веб-интерфейс.**

**Главное отличие:** проект = репозиторий. Контекст формируется файлами и правилами внутри него.

### Claude Code

    # В корне репозитория
        CLAUDE.md              # правила проекта, автоматически подтягиваются
        knowledge/             # документация, примеры
        .clinerules            # правила для Cline (если используешь)

**Тактика:**

- `CLAUDE.md` автоматически подтягивается в контекст — идеален для правил проекта
- Можно ссылаться на файлы через `@`, чтобы точечно давать контекст
- Knowledge живёт в `docs/` или `knowledge/`

### Cursor

    # В репозитории
        .cursor/
          └── rules            # правила проекта, версионируются в git

**Смысл:** ты буквально упаковываешь `01-rules.md` в правила редактора.

### [GitHub Copilot](https://github.com/features/copilot) + VS Code

- **Repository custom instructions** — чтобы Copilot понимал, как собирать, тестировать и менять проект
- **Workspace context** — чтобы чат понимал кодовую базу целиком
- **\# mentions** — можно явно добавлять контекст (файлы, папки, символы)

### [Cline](https://marketplace.visualstudio.com/items?itemName=saoudrizwan.claude-dev) (VS Code)

Управление контекстом построено вокруг явных ссылок на файлы/папки и правил типа `.clinerules` или "memory bank".

**Вывод:** даже в "кодовых" LLM проект настраивается теми же 5 файлами. Просто они живут в репозитории, а не отдельной папке.

## Проверьте понимание: Структура проекта

- **Прогресс:** 1 / 2
- **Вопрос 1:** Из скольких основных частей состоит проектный контур?
- **Варианты ответа:**
  - 4 частей: цель, контекст, примеры, инструкции
  - 2 частей: правила и примеры
  - 3 частей: правила, знания, промпты
  - 5 частей: brief, rules, knowledge, templates, decisions-log

## Проект vs промпт

| Критерий | Промпт | Проект |
|----|----|----|
| **Контекст** | Каждый раз вручную | Автоматически подтягивается |
| **Правила** | Копируешь в каждый чат | Живут в инструкциях проекта |
| **Знания** | Вставляешь текстом | Файлы в базе знаний |
| **История** | Разбросана по чатам | Вся внутри проекта |
| **Переиспользование** | Копируешь промпт | Используешь проект заново |
| **Эволюция** | Нужно обновлять везде | Обновил правила — работает везде |

## Когда использовать проект

**Используй проект:**

- Работаешь над задачей больше недели
- Нужны одни и те же правила/термины
- Команда использует общий контекст
- Есть документация, которую нужно подтягивать

**Промпта достаточно:**

- Разовая задача на 1-2 часа
- Контекст помещается в 1 промпт
- Нет повторяющихся правил
- Не планируешь возвращаться к задаче

## Чек-лист настройки проекта

### За 15 минут

1.  Создай папку `project-name/` (или Project в LLM)
2.  Напиши `00-brief.md` (цель, аудитория, критерии)
3.  Напиши `01-rules.md` (формат, стиль, запреты)
4.  Собери `02-knowledge/` (2-3 ключевых документа)
5.  Загрузи в выбранный инструмент

**Результат:** ассистент понимает контекст без объяснений в каждом чате.

## Частые ошибки

### ❌ Ошибка 1: Слишком общие правила

**Плохо:** "Будь полезным"

**Хорошо:** "Формат ответа: тезис → пример → вывод. Длина: не больше 3 абзацев"

### ❌ Ошибка 2: Забыл про базу знаний

Правило написано в одном чате, но не добавлено в базу знаний → в следующем чате ассистент его не знает.

### ❌ Ошибка 3: Не обновляешь правила

Если правило не работает — обнови `01-rules.md` и перезагрузи в проект. Иначе будешь повторять одно и то же в каждом чате.

### ❌ Ошибка 4: Храним всё в одном проекте

Разные задачи = разные проекты. Не смешивай "код ревью" и "написание статей" в одном контейнере.

## Итог

**Проект — это структура, а не фича интерфейса.**

Минимум:

- Папка на компьютере с 5 файлами
- Или репозиторий с `CLAUDE.md` и `knowledge/`
- Или проект в ChatGPT/Claude/Яндекс GPT с инструкциями и файлами

**Результат:** ассистент работает предсказуемо, контекст не теряется, правила эволюционируют.
